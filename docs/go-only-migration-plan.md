# BigClaw Go-Only Migration Plan

This plan is generated by `bigclawctl go-migration plan` and is the repo-native control surface for retiring the remaining Python and non-Go implementation paths.

## Acceptance

- Maintain `main` as Go-first while retiring remaining Python and non-Go runtime, script, test, and toolchain surfaces in batches.
- Keep the migration inventory explicit so every remaining non-Go path has an owner and a retirement lane.
- Run at least 10 parallel slices under Symphony without forcing one large cutover PR.

## Branch And PR Strategy

- Branch naming: `symphony/<slice-id>`
- PR title template: `BIG-VNEXT-GO-xxx: <slice title>`
- Keep slices branch-local and small enough to merge independently into `main`.
- Do not mix runtime migration, test migration, and toolchain retirement in one PR unless a later slice depends on the previous merge.
- Use the generated slice identifiers as the stable Symphony branch and tracker references.
- Every slice must run targeted `go test` commands for the touched Go packages.
- Tracker closeout must record exact commands and pass/fail results, not generic statements.
- Cutover-facing slices must also rerun `go test ./...` from `bigclaw-go` before merge.
- Do not delete Python or shell surfaces until an equivalent Go-owned runtime or validation path exists.
- Prefer staged retirement: generate parity evidence, merge the Go replacement, then freeze and remove the legacy path.
- Treat `main` as the only evolving branch; all migration branches rebase on current `main` before merge.
